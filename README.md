This document provides an exhaustive technical mapping of the Conversational AI Assistant. ## 🚀 Getting Started

Follow these steps to set up and run the application on your local machine.

### 📋 Prerequisites
- **Python 3.10+**
- **Node.js & npm**
- **PostgreSQL** (Database named `Registrations` with user `postgres`)
- **Redis** (For Celery background tasks)
- **Piper TTS** (Installed at `C:\piper\`)

1. **Navigate to the root directory:**
   ```powershell
   cd "D:\AVIOX PVT LTD\AI_Model\Annie'sUI"
   ```

## ⚙️ 1. Backend Architecture (`/backend/`)

The backend functions as the central nervous system of the application. It handles audio transcription, semantic memory storage, language generation, and speech synthesis.

### 🧠 A. AI Models Used (Backend Pipeline)
These are the core Artificial Intelligence engines that process the conversational data.

| Model / Technology | Location in Codebase | What it means / What it does |
| :--- | :--- | :--- |
| **Gemini-2.0-Flash** | `backend/voice_assistant.py` | **Reasoning Option 1 (Default):** A Large Language Model by Google. It powers the "Senior Companion" persona with high speed and reliability. |
| **Llama-3.3-70B** | `backend/voice_assistant.py` | **Reasoning Option 2 (Groq):** An ultra-fast, high-capability model served via Groq. Excellent for handling complex emotional conversations and patient teaching roles. |
| **OpenAI Whisper (Medium)**| `backend/voice_assistant.py` | **Speech-to-Text (STT):** Translates the user's spoken words into text. It is specially trained to handle different cadences and vocal patterns. |
| **Piper (en_US-lessac-high)**| `backend/voice_assistant.py` | **Text-to-Speech (TTS):** Converts the AI's empathetic text responses back into a warm, natural-sounding voice file. |
| **all-MiniLM-L6-v2** | `backend/voice_assistant.py` | **Semantic Memory:** Used to convert memories into numerical vectors for long-term recall in Qdrant. |

### 🛠️ B. Core Technologies (Backend Pipeline)
These are the foundational frameworks and databases that support the AI models.

| Technology | Location in Codebase | What it means / What it does |
| :--- | :--- | :--- |
| **FastAPI** | `backend/main.py` | **Web Framework:** The highly asymmetrical Python HTTP server. It manages the `/chat/audio` and `/chat/text` endpoints that the frontend talks to. |
| **PostgreSQL & psycopg2** | `backend/main.py` & `migrate_db.py` | **Relational Database:** Stores user credential data securely. Crucially, it tracks `role` assignments (User vs. Admin) to enforce Role-Based Access Control (RBAC). |
| **Qdrant (Vector DB)** | `backend/voice_assistant.py` (Line 115) | **Long-Term Memory Storage:** A database specifically designed to store the high-dimensional numbers generated by `all-MiniLM-L6-v2`. It acts as the assistant's permanent memory bank. |
| **LangChain** | `backend/voice_assistant.py` (Line 242) | **AI Orchestrator:** The glue code. It grabs the user's text, grabs the Qdrant memory, injects it all into a massive `ChatPromptTemplate`, and fires it to the Gemini model. |

### 🎛️ C. Parameters for Fine-Tuning (Backend)
These configuration variables directly control the AI's behavior, responsiveness, and memory retrieval logic.

| Parameter | Location | Meaning / Effect on System |
| :--- | :--- | :--- |
| `VAD_THRESHOLD` (0.6) | `backend/voice_assistant.py` (Line 43) | **Voice Detection Sensitivity:** A score from 0.0 to 1.0. Determines how confident the local terminal mic must be that a sound is "human speech" before it starts recording. |
| `SILENCE_LIMIT` (2.0s) | `backend/voice_assistant.py` (Line 44) | **Sentence Break Timer:** How many seconds of absolute silence the terminal mic waits before deciding the user is finished speaking and cutting off the recording. |
| `SPEECH_LIMIT` (15.0s) | `backend/voice_assistant.py` (Line 45) | **Maximum Recording Cap:** A hard fail-safe limit. Prevents the terminal mic from recording indefinitely if background noise prevents a true silence. |
| `RECENT_TURNS_TO_INJECT` (6) | `backend/voice_assistant.py` (Line 66) | **Short-Term Memory Cache:** The exact number of recent back-and-forth messages that are forcefully injected into the AI's prompt to maintain immediate context without needing a database search. |
| `QDRANT_SEARCH_LIMIT` (8) | `backend/voice_assistant.py` (Line 67) | **Long-Term Recall Breadth:** The max number of historical memories Qdrant will fetch. A higher number gives the AI more background context, but slows down inference slightly. |
| `QDRANT_SCORE_THRESHOLD` (0.2) | `backend/voice_assistant.py` (Line 68) | **Semantic Leniency:** How "mathematically similar" a past memory must be to the current prompt to be retrieved. `0.2` is very lenient, allowing the AI to successfully recall loosely related facts (like food preferences). |
| `RECENCY_WEIGHT` (0.01) | `backend/voice_assistant.py` (Line 69) | **Memory Tie-Breaker:** An algorithmic multiplier. If two memories are equally relevant, this weight mathematically forces the system to trust the chronologically *newer* memory. |
| `INTERRUPT_SENSITIVITY` (0.03) | `backend/voice_assistant.py` (Line 50) | **AI Cutoff Threshold:** Measures raw microphone volume (RMS). If the user begins speaking louder than this threshold *while* the AI is still talking, the AI instantly aborts its TTS playback and begins listening. |

## 🎨 2. Frontend Architecture (`/frontend/src/`)

The frontend is the visual and interactive layer. It manages the continuous browser microphone loop, UI rendering, and user authentication state.

### 🧠 A. Models & Scripts Used (Frontend Pipeline)
Unlike the backend, the frontend relies on browser APIs and JavaScript scripts rather than heavy neural networks.

| Model / Script | Location in Codebase | What it means / What it does |
| :--- | :--- | :--- |
| **Hark (VAD Script)** | `frontend/src/pages/Chat.jsx` (Line 141) | **Volume Activity Detector:** A lightweight JavaScript module that constantly samples the browser's microphone decibels. It fires `speaking` and `stopped_speaking` javascript events, which drive the UI animations and the Continuous Voice Mode timing. |
| **MediaRecorder API** | `frontend/src/pages/Chat.jsx` (Line 121) | **Browser Audio Engine:** A native HTML5 API. It captures the raw hardware audio chunks from the user's mic and automatically compresses them into an efficient `.webm` Blob file format for transmission. |
| **HTML5 Audio API** | `frontend/src/pages/Chat.jsx` (Line 231) | **Audio Playback Engine:** A native browser class (`new Audio()`) used to stream and play the final `.wav` file response generated by the backend TTS system. |

### 🛠️ B. Core Technologies (Frontend Pipeline)
The foundational frameworks driving the UI experience.

| Technology | Location in Codebase | What it means / What it does |
| :--- | :--- | :--- |
| **React 18 & Vite** | `frontend/src/App.jsx` + `main.jsx` | **Component Framework:** The core library used to build the interactive UI components. Vite is the incredibly fast build tool that bundles the JSX files for the browser. |
| **React Router DOM** | `frontend/src/App.jsx` | **Navigation Manager:** Handles "Route Guarding". It checks if a `user` has an `admin` role; if they do, it allows them to see the `/` AdminDashboard. If they are a standard user, it locks them to the Chat interface. |
| **Vanilla CSS** | `frontend/src/index.css` & `Chat.css` | **Design System:** Custom CSS variables defining a "Lively Beige & Deep Navy" theme. Uses strictly custom layout properties (like `flex: 1`) to ensure full-window scrolling and sticky input fields. |

### 🎛️ C. Parameters for Fine-Tuning (Frontend)
These logic hooks and variables control the exact feel of the Voice Mode in the browser.

| Parameter | Location | Meaning / Effect on System |
| :--- | :--- | :--- |
| `threshold: -50` | `frontend/src/pages/Chat.jsx` (Line 142) | **Decibel Trigger Limit:** Specific to Hark. If the user's microphone volume falls below -50dB, Hark assumes the user is not speaking or there is only background hum. If volume exceeds -50dB, it triggers the `speaking` visual UI pulse. |
| `silenceTimeout (1500ms)` | `frontend/src/pages/Chat.jsx` (Line 165) | **Browser Silence Cutoff:** The absolute timer for the UI mic. Once Hark fires a `stopped_speaking` event, this timer counts `1.5` seconds. If the user doesn't start speaking again before the timer expires, the UI kills the mic, finalizes the `.webm` Blob, and submits it to the API. |
| `echoCancellation: true` | `frontend/src/pages/Chat.jsx` (Line 117) | **Hardware Audio Filter:** An instruction forcefully sent to the operating system's mic. Prevents the physical microphone from "hearing" the audio playing out of the user's speakers, which stops infinite feedback loops. |
| `noiseSuppression: true` | `frontend/src/pages/Chat.jsx` (Line 117) | **Hardware Static Filter:** An instruction to the OS to digitally scrub background static (fans, AC units) from the `getUserMedia` stream before the MediaRecorder compresses it. |
| `isContinuousVoiceRef` | `frontend/src/pages/Chat.jsx` (Line 22) | **Autonomous Loop State:** A React Reference Boolean. If true, the moment the backend API finishes generating a response and sending it down, the frontend will automatically trigger `startListening()`, throwing the user right back into the mic loop without needing a physical click. |
| `currentAudioRef.current.pause()` | `frontend/src/pages/Chat.jsx` (Line 158) | **Frontend Interruption Hook:** The logic that stops the AI. Even while the `HTML5 Audio` API is playing out loud, Hark is still running. If Hark detects speech, it immediately calls `.pause()` on the Audio API, terminating the AI's sentence mid-breath. |

## 🔐 4. Authentication & User Management

The application features a robust, dual-strategy authentication system and dynamic user profile management, deeply integrated across the frontend and backend.

### 🔑 A. Dual Authentication Flows

| Strategy | Location | How it Works |
| :--- | :--- | :--- |
| **Email & Password** | `backend/main.py` (`POST /login`) | Standard credential verification. Hashes are securely compared against the `users` table in PostgreSQL. Returns an authenticated user payload including their system `role`. |
| **Google SSO (OAuth2)** | `backend/main.py` (`POST /google-login`) | Utilizes `@react-oauth/google` on the frontend to generate a token, which is securely verified on the backend using the `google-auth` Python library. |
| **Smart Registration** | `backend/main.py` (`POST /google-login`) | When a Google SSO user logs in for the first time, the backend automatically intercepts the flow. It generates a random password, creates a username, and inserts a new row into the `users` database. |
| **Phone Number Intercept** | `frontend/src/pages/Login.jsx` (Line 80) | If the backend detects a missing `phone` number during a Google login, it flags `requires_phone: true`. The frontend visually intercepts the login to prompt the user for their phone number before granting entrance. |

### 👤 B. Profile & Data Management

| Subsystem | Location | What it means / What it does |
| :--- | :--- | :--- |
| **Unified Payload (`GET /profile`)** | `backend/main.py` (Line 304) | A highly efficient `LEFT JOIN` SQL query connects the `users` table containing core identity (email, name, role) directly to the `addresses` table (city, state, pincode), returning a massive merged dictionary. |
| **Dynamic `COALESCE` Updates** | `backend/main.py` (`PUT /profile`) | An advanced `UPDATE` strategy. By wrapping SQL values in `COALESCE(%s, column_name)`, the endpoint allows partial, asymmetrical data updates without accidentally wiping out existing unmodified fields in the database. |
| **Profile Display UI** | `frontend/src/components/ProfileModal.jsx` | A dedicated React overlay component accessible from the `Layout` Navigation bar. It seamlessly switches between a "Read-Only" display state and an "Edit" form state. |
| **Dynamic Avatar Serving** | `backend/main.py` & Layout UI | When a user uploads a custom profile picture, the backend securely writes it to `/static/profiles` and saves the exact `http://` URL strings into the `profile_image` PostgreSQL column. The frontend instantly retrieves this string during login to render a personalized `<img src={...}/>` directly in the navigation bar. |
